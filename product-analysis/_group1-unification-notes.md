# Group 1 — cross-cutting unification notes

Threads I noticed across diff-view, file-sidebar, full-file-view, context-expansion, and keyboard-help.

- **Read-coverage is computed in three places, none reused.** `hunkCoverage` / `fileCoverage` (`state.ts:1456-1491`) feed the diff hunk-header %, the sidebar meter, and the status bar `read NN%`. Each consumer re-walks `readLines`. One memoised "coverage projection" per (cs, readLines) tick would let all three read from the same shape, alongside future per-file AI-note counts.
- **Mode toggles live as two `Set<fileId>` with reducer-enforced mutual exclusion.** `fullExpandedFiles` + `previewedFiles` (`types.ts:285-290`) should be a single `Record<fileId, "diff" | "source" | "preview">`. Removes the inverse-set logic in `TOGGLE_EXPAND_FILE` / `TOGGLE_PREVIEW_FILE` (`state.ts:511-528`). Full-file mode and preview mode both become rows in one map.
- **Source mode silently disables cursor + mouse selection** because `FullFileView`'s lines carry no `data-line-idx` (`DiffView.tsx:802-823`). Either unify Source rendering with the per-hunk renderer (one row template, `fullContent` is "one giant hunk") or document the limitation in the mode toggle UI. Today it's an invisible cliff.
- **`FullFileLineViewModel` is `DiffLine` + a precomputed sign.** The sign is `kind === "add" ? "+" : kind === "del" ? "-" : " "`. Either inline that in the renderer or push it onto `DiffLine` once. Same applies to `aiGlyph` on `DiffLineViewModel`.
- **The comment-column conditional (`commentColOn`) threads through 5 components.** `Line`, `ContextLine`, `FullFileView`, `HunkLinesBlock`, `ContextLinesBlock` (`DiffView.tsx`). One CSS variable on the diff root would replace the prop chain.
- **`expandContext.ts` and the full-file view derive from the same `postChangeText`** but go through `expandAbove/Below` and `fullContent` as separately stored fields on `DiffFile`. A rebuild could keep only `postChangeText` and derive both views on read. Saves ~150 lines and unifies the hydration story.
- **Help-overlay context hints duplicate KEYMAP knowledge.** `buildHelpContext` (`ReviewWorkspace.tsx:2663-2737`) hard-codes chord strings (`⇧m`, `]/[`, `⌘k`); the status bar `DEFAULT_HINT` (`view.ts:617`) does the same. A `keysForAction(action)` helper backed by KEYMAP would let both reuse the registry.
- **The gutter-glyph legend in `HelpOverlay.tsx:139-163` and `aiGlyph` in `view.ts:339-352`** encode the same severity→glyph map twice. One shared `severityGlyph` constant covers both.
- **Diff view and Sidebar each compute file-scoped interaction stats independently.** `buildCommentCounts` (`view.ts:466-495`) walks `state.interactions` to bucket by file; `buildCommentStops` (`state.ts:1031-1076`) walks again to build the n/N nav list; per-line glyph picking walks AI-note lookups per hunk. One per-file projection from `selectInteractions` would replace all three.
- **Expand-context state isn't persisted** (`persist.ts:73-84` omits `expandLevelAbove`/`expandLevelBelow`) while every other progress field is. Reviewers lose every revealed block on reload. Cheap to add to the snapshot.
- **`KEYMAP.find` priority ordering is load-bearing but untested.** Reorder regressions (`Shift+ArrowDown` vs `ArrowDown`, `n` with vs without `hasSuggestion`) would slip silently. Two snapshot tests would lock the contract.
- **`onPickFile` and `onJumpToFirstComment` both want a cursor.** The Sidebar emits a `fileId` and the orchestrator resolves to `Cursor`. Pre-computing the target `Cursor` in `SidebarFileItem` removes one indirection per click handler.
