# Group 2 — cross-cutting unification notes

Scope: review-progress, sign-off, session-persistence.

- **The headline split.** Interactions live in SQLite (`server/src/db/`); review progress (cursor, readLines, reviewedFiles, reviewedChangesets, dismissedGuides, drafts, quiz) lives in localStorage (`web/src/persist.ts`). The split is mechanically explained — agents and the MCP bridge can't see localStorage — but it's also a relic of the pre-server prototype. The server is now a hard dependency (`docs/architecture.md:8`), so the "browser-only fallback" justification is gone.
- **Everything except drafts and UI prefs wants to move.** `cursor`, `readLines`, `reviewedFiles`, `reviewedChangesets`, `dismissedGuides`, and `quiz` (Q&A log) are all session state that would benefit from server ownership: multi-window safety, cross-machine continuity, MCP/agent readability, audit history. Drafts and UI prefs (`theme`, `inspector`, `inline-comments`, zoom) stay local — loss-tolerant, machine-specific, intentionally so.
- **Sign-off is the highest-leverage move.** Once `reviewedFiles` and `reviewedChangesets` are server-side, an agent can ask "did the reviewer sign this off?" cleanly. Today they can't — see `docs/features/sign-off.md:60` ("no server endpoint or agent acts on it"). The IDEA doc lists agent-coordinated review as a target; storage is the gating constraint.
- **One bundle per changeset on load.** `LOAD_CHANGESET` already triggers a per-changeset interactions fetch (`App.tsx:263–281`). A rebuild can collapse progress + interactions + sign-off into one server bundle, killing the in-flight-fetch / StrictMode workaround comments at `App.tsx:270–280`.
- **Schema-versioning belongs server-side.** `persist.ts` runs a no-migration policy (v3→v7) on the client. Versioned migrations in SQL are routine; versioned blobs in localStorage with "fail closed on mismatch" is a prototype shortcut that becomes friction at scale.
- **`reviewedChangesets` token-list compaction.** The accumulating-tokens shape is fine in principle but unbounded; a SQL row-per-token with `signed_off_at` makes pruning a one-liner and lets the UI show sign-off history (which has product value: "you've signed off three times on this PR as it evolved").
- **`recents.ts` is rightfully separate.** It's browser history, not session state; the dedicated key + cap of 5 is the boring correct shape. Keep it that way in a rebuild.
- **Per-worktree live-reload toggle is correctly local-only** (`persist.ts:26`) — outlives any session.
- **Stale doc reference to investigate.** `docs/architecture.md:194` cites `web/src/interactionViewMode.ts` but the file isn't in the tree. Either the doc lags or the persisted-mode toggle quietly disappeared; either way it's a loose end touching session persistence.
